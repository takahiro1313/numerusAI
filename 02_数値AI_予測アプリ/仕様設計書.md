## 1. システム構成

### 1.1 全体構成図

```
┌──────────────────────────────────────────────┐
│ 参加者ブラウザ                                │
│                                              │
│  Streamlit UI                                │
│   ├─ 入力：担当×月の時系列プレビュー    │ 🔒 見るだけ（Step1）
│   ├─ 編集セル：fit の穴埋め → [学習ボタン]    │ 🟢 触る（Step2）
│   ├─ 編集セル：predict の穴埋め → [推論ボタン]│ 🟢 触る（Step3）
│   ├─ パラメータ：n_estimators / max_depth    │ 🟢 触る（Step4）
│   └─ 出力：予測成約数 / 重要度 / train・val対比 │ 🔒 結果表示
│                                              │
└───────────────────┬──────────────────────────┘
                    │ 「学習/推論」押下（編集セルの文字列を送信）
                    ▼
   ┌───────────────────────────────────────┐
   │ Streamlit サーバ（Python）            │
   │  - ダミーCSV読込 → train/val 分割      │
   │  - 限定名前空間で exec（fit/predict）  │
   │  - XGBoost 学習・推論（数秒）          │
   │  - train/val スコア・特徴量重要度算出  │
   │  - 結果を st.session_state に格納       │
   └───────────────────────────────────────┘

  ※ 外部API・認証なし。計算はサーバ側ローカルのみ。
  ※ 1人1URL = 5インスタンスを別々にデプロイ（相互干渉なし）。
```

### 1.2 構成要素

| 要素 | 役割 | 技術 |
|---|---|---|
| Streamlit サーバ | 画面提供 / データ管理 / 学習・推論の安全実行 | Python, Streamlit |
| 数値AIモデル | 先行活動→翌月成約 の学習・推論 | XGBoost（`XGBRegressor`） |
| 編集セル | 参加者が `fit`/`predict` を書くUI | `st.text_area` |
| パラメータUI | `n_estimators`/`max_depth` の変更 | `st.slider` |
| 出力領域 | 予測値・スコア対比・重要度グラフ | `st.metric` / `st.dataframe` / `st.bar_chart` |

### 1.3 設計の要点（なぜこの構成か）

- **書く体験を fit/predict に絞る**：穴埋めは2箇所のみ。データ読込・スコア計算・グラフ描画は 🔒 固定領域。
- **過学習を体感させる**：学習時に train/val を分割し、両スコアを並べて表示。パラメータを上げると訓練↑・検証↓を見せる（スライド54）。
- **その場で数秒学習**：数百行のダミーCSV＋XGBoostで、ボタン押下から結果まで体感即時。
- **ITパートと同じUI言語**：[学習ボタン]/[推論ボタン]は IT の「押す→処理が走る」の使い回し。演習用/正解用・🔒/🟢 表記も IT と統一。

## 6. 安全な実行（exec）設計

参加者が書く `fit`/`predict` をサーバ側で実行するため、ITパートと同方針で事故防止を徹底する（HANDOFF §3.1 / §9）。

| 項目 | 方針 |
|---|---|
| 名前空間の限定 | `exec(code, safe_globals, ns)` で**渡す変数・関数を絞る**（`model`/`X_train`/`y_train`/`X_new` 等のみ）。`__builtins__` は最小化 |
| I/O 遮断 | ファイル/ネットワークI/Oは渡さない。`import os` 等が無くても完結 |
| 例外捕捉 | `try/except` で全例外を捕捉し、**画面にエラーメッセージを表示**（アプリは落とさない） |
| 前提 | 信頼できる少人数（役員）＋インスタンス分離済み。ただし上記は必須 |

```python
def run_fit(code, ns):
    try:
        exec(code, {"__builtins__": SAFE_BUILTINS}, ns)  # ns に model, X_train, y_train
        model = ns["model"]
        train = score(model, ns["X_train"], ns["y_train"])
        val   = score(model, ns["X_val"],   ns["y_val"])
        return train, val
    except Exception as e:
        return None, f"⚠️ エラー: {e}（fit の書き方を見直してください）"
```

## 13. 推論エンドポイント（PdM FB 2026-06-19・生成AI連結）

数値AIの予測を**まーぴー（生成AI）が呼んで話す**ための導線。学習は従来どおりStreamlitアプリ内、推論を外から叩けるようにする。

### 13.1 方式の選択肢

| 方式 | 概要 | 難易度 | 対応 |
|---|---|---|---|
| **A. 別エンドポイント（FastAPI 等）** | 学習済みモデルを保存し、`POST /predict`（活動量→翌月成約）を別サービスで提供 | 中 | **ベスト**（まーぴーが直接呼ぶ） |
| **B. Streamlit内の簡易API** | クエリパラメータ等で予測を返す軽量経路を同アプリに追加 | 中 | ベスト寄り（構成はシンプル） |
| **C. 出力CSV共有** | 学習後に予測結果CSVを書き出し、まーぴーがそれを読む | 低 | **ベター**（最低ライン） |

> **当日方針（確定 2026-06-19）：まず C（CSV読み）を確実に。** A/B（エンドポイント）は余力があれば。
> ＝学習はStreamlit上、学習後に予測結果CSVを書き出し、まーぴーがそれを読んで話す。

### 13.2 推論I/O（共通イメージ）

- 入力：活動量（特徴量）`{問い合わせ対応数, 初回面談数, 面談数, 企業評価数, 案件提案数}`
- 出力：`{予測成約数: float}`
- モデルは「学習済みの状態」を前提（事前学習 or デプロイ済アプリで学習したものを保存）。

### 13.3 まーぴー側の呼び出し（連結）

- ベスト：まーぴーが上記エンドポイントへリクエスト → 返った予測成約数を音声で答える。
- ベター：まーぴーが出力CSVの該当行を読む → その値を音声で答える。
- 詳細は `../03_生成AI_まーぴー/要件定義書.md §7.3`。
